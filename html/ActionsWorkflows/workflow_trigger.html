Control workflow triggers

When creating a workflow, the second line, after the flow's name, is the trigger.
name: some name
  on: push

There are many events that can trigger an event. Another reference on the variety of action triggers. The two most common are:

  • push
  • pull-request

These events can also be paired with specific branches
on:
  push:
    branches:
      - master
      - 'devel-html'
OR exclude certain branches
  pull_request:
    branches-ignore:
      - 'other'
OR use pattern matching for branch names
  pull_request:
    branches-ignore:
      - 'devel-**'

Another way to trigger a workflow is on a definite chronological schedule. In this code the values 1-5 indicate different measures of time. The value shown here will prompt the the execution on Friday April 3 at 2:01am UTC. on:
  schedule:
    - cron: '1 2 3 4 5'

  1. Minutes (0-59)
  2. Hour (0-23)
  3. Day (1-31)
  4. Month (1-12)
  5. Weekday (0=Sunday through 6=Saturday)

Another option is to execute the job on a periodic schedule. The '*' is a wildcard with '*/1' indicating any time that matches modulo 1. See this Wikipedia article for more information about the exact syntax. This one executes every 6 hours so 0000Hrs, 0600Hrs, 1200Hrs, and 1800Hrs all UTC. on:
  schedule:
    - cron: '0 */6 * * *'
 This one executes every day at midnight eastern time. Remember the time difference with UTC. on:
  schedule:
    - cron: '0 5 * * */1' This one executes every two hours on weekdays from 4am EST to 6pm EST. This will likely be used to run the same job periodically during the workday. on:
  schedule:
    - cron: '0 9,11,13,15,17,19,21,23 * * 1-5'

By employing a group of these triggers you can specify a test to run under specific conditions and no other. There are a number of different combinations for full documentation see this page.
